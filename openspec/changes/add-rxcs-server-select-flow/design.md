## Context

RXCS 当前已经完成启动、弹窗处理与登录流程，登录成功后下一步就是进入正确区服。当前目标区服已明确为“绿色一区”，但项目尚未建立登录后选区页的稳定识别与点击方案，因此必须把选区能力独立建模，而不是临时把点击逻辑硬写进主流程。

约束条件如下：
- 选区流程必须建立在登录成功之后。
- 目标区服已明确：绿色一区。
- 选区页可能涉及绿色版入口和一区按钮两个阶段动作，不能视为单一步骤。
- 若控件无法稳定识别，允许使用固定命名的图片资源作为兜底。
- 后续主页和任务流都依赖正确区服，因此选区结果必须校验。

## Goals / Non-Goals

**Goals:**
- 建立 RXCS 登录后选区能力的正式边界与实现路径。
- 明确绿色版入口与绿色一区选择的分步处理逻辑。
- 定义选区原子测试模式，支持逐步验证。
- 为后续主页识别和任务流提供稳定入口。

**Non-Goals:**
- 本次不实现进入区服后的主页任务流。
- 本次不处理多区服配置管理。
- 本次不覆盖所有颜色版块与所有大区，只针对当前目标区服。

## Decisions

### 1. 选区流程拆成两个阶段
- 阶段一：点击绿色版入口
- 阶段二：选择绿色一区

原因：这两个动作很可能处于不同的界面状态，必须分步验证，不能混成一次点击。

备选方案：
- 直接把“绿色版 + 绿色一区”写成一次流程：调试粒度太粗，失败时难定位。

### 2. 选区流程继续遵守“控件优先，图片兜底”
- 若页面存在可稳定识别控件，优先走控件
- 若页面缺少稳定控件，则用图片资源识别绿色版入口和绿色一区

原因：沿用当前项目总体规则，减少后续维护成本。

### 3. 选区必须有独立 ready 检测
在点击任何区服按钮前，必须先确认当前页面已进入选区页。

原因：如果登录成功后存在过场页、公告页、加载页，直接点选区会把问题混在一起。

### 4. 选区结果必须校验
点击绿色一区后，至少要有一种结果确认：
- 区服选中态出现
- 选区页消失
- 进入主页特征出现

原因：如果只点不验，后续流程可能在错误区服或未进区状态下继续运行。

## Risks / Trade-offs

- [选区页控件可能不稳定] → 预留图片资源兜底，先实现原子测试模式。
- [绿色版入口和绿色一区都可能有多个视觉状态] → 资源命名固定，必要时区分入口图、选中图、进入后特征图。
- [登录后可能出现中间页] → 先实现 server page ready，再允许点击区服。

## Migration Plan

1. 新增选区 change，不污染已归档的登录 change。
2. 先实现 `server_page_ready`。
3. 再实现 `server_click_green_entry_only`。
4. 再实现 `server_click_green_zone_only`。
5. 最后整合为完整选区流程。
6. 选区成功后，再进入主页与任务流设计。
