## Context

当前 RXCS 已经完成 AutoJS6 项目初始化，并已迁入一部分公共模块与启动流程，但登录能力仍未正式落地。现阶段已确认登录页可通过控件 `id` 识别用户名输入框、密码输入框与登录按钮；协议勾选状态仍需依赖图片资源兜底。此前按键精灵方案在登录页反复暴露出输入框污染、调试粒度过粗、工具行为不透明等问题，因此本次设计必须优先保证流程可调试、可验证、可扩展。

约束条件如下：
- 登录页必须优先使用 AutoJS6 控件能力，不再沿用按键精灵式的过程脚本思维。
- 登录流程必须支持原子级测试，先验证单步动作，再整合为完整链路。
- 协议勾选状态当前没有稳定的控件语义，需保留图片识别兜底。
- 登录按钮允许通过控件 `id` 直接触发，不再要求以图片作为唯一入口。
- 后续选区、日常任务等流程都依赖登录能力，因此登录模块必须清晰分层。

## Goals / Non-Goals

**Goals:**
- 建立 RXCS 登录页的正式实现边界与调试模型。
- 明确用户名、密码、协议、登录按钮各自的识别与触发策略。
- 提供原子测试模式，支持逐步验证登录页动作。
- 定义输入与校验的失败即停原则，避免错误状态继续扩散。
- 为后续选区与任务流程提供稳定前置入口。

**Non-Goals:**
- 本次不覆盖选区逻辑。
- 本次不定义登录成功后的完整首页状态机。
- 本次不处理账号体系、认证体系、远程配置等更上层业务。
- 本次不为所有后续页面一次性补齐全部图片资源。

## Decisions

### 1. 登录页采用“控件优先，图片兜底”
- 用户名输入框：使用 `id("huanyu_defaultlogin_et_user")`
- 密码输入框：使用 `id("huanyu_defaultlogin_et_pwd")`
- 登录按钮：使用 `id("huanyu_defaultlogin_bt_login")`
- 协议勾选状态：使用 `agree_no.png` / `agree_yes.png`

原因：当前页面已确认上述三个核心控件存在稳定 `id`，继续强行用图片处理控件输入只会增加不确定性。协议勾选因为当前更容易通过视觉资源区分状态，所以保留图片兜底更稳。

备选方案：
- 全部走图片：实现简单，但对输入框和按钮的可维护性较差。
- 全部走控件：理想但协议状态当前未验证稳定，风险更高。

### 2. 登录流程必须拆为原子测试模式
测试模式至少包含：
- login_page_ready
- login_input_user_only
- login_input_pwd_only
- login_agree_only
- login_click_only
- login_full

原因：当前项目已经证明，整条链路一起调试会掩盖根因。必须先把页面 ready、输入、协议、点击分别测通，再整合。

备选方案：
- 直接跑完整登录：调试成本太高，已被证明不可取。

### 3. 输入流程采用“动作 + 回读校验”
用户名与密码输入都必须遵守：
1. 定位目标控件
2. 点击目标控件
3. 执行内置输入
4. 回读目标控件文本或可验证状态
5. 若目标未变化或其他输入框被污染，立即失败

原因：仅凭“调用过输入函数”不能证明输入成功，必须有回读校验。

备选方案：
- 只执行点击和输入，不校验：会导致错误状态扩散。

### 4. 协议处理采用显式状态判断
实现顺序：
1. 先找 `agree_yes.png`
2. 若已勾选，直接通过
3. 否则找 `agree_no.png`
4. 点击后再次检查 `agree_yes.png`
5. 若仍未变为勾选态，则判定失败

原因：协议是登录前的强前置条件，必须显式验证状态切换，不能只凭点击动作继续流程。

### 5. 登录按钮点击后必须有结果校验
第一版结果校验至少支持以下任一条件：
- 登录页消失
- 登录页关键控件消失
- 进入后续页面的特征出现

原因：只点击登录按钮而不判断结果，会让后续流程在未登录状态下继续执行。

## Risks / Trade-offs

- [输入框控件虽可识别，但输入行为可能仍受页面实现影响] → 通过原子测试模式先独立验证用户名输入与密码输入。
- [协议截图质量不稳定会影响识别] → 统一截图规范，只裁勾选控件本体并固定命名。
- [登录结果校验不足会导致误判] → 第一版至少要求登录页特征消失，后续再补主页特征图。
- [控件与图片混用会增加实现复杂度] → 仅在必要处使用图片，减少混用范围。

## Migration Plan

1. 在现有 AutoJS6 项目中新增登录模块，不改动已可运行的启动与弹窗处理骨架。
2. 先实现 `login_page_ready` 与输入框单项测试。
3. 补齐 `agree_no.png` 与 `agree_yes.png` 后实现协议测试。
4. 接入登录按钮点击与登录结果校验。
5. 单项测试全部通过后，再启用 `login_full`。
6. 若某一步不稳定，回滚到对应原子测试，不继续推进整条链路。
